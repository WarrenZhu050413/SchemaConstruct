Git Commit Objects & Merging Internals

üîë Key Concept: Every Commit Points to a Tree

Commits don't store files‚Äîthey store pointers! Each commit object contains:

  • tree: Hash of the root directory snapshot
  • parent(s): Hash(es) of previous commit(s)
  • author/committer: Who made the commit
  • message: The commit description

Part 2: How Merging Works

Merge Commits Have TWO Parents

A normal commit has one parent. A merge commit has two (or more) parents‚Äîone from each branch being merged.

Scenario: Merging feature-branch into main

üîë Key Insight: The Merge Commit

  • TWO parents: Points to BOTH branch tips
  • ONE tree: The merged result (combined state)
  • Git uses the two parents to reconstruct the merge history
  • The tree shows what the files look like after merging

Step-by-Step: What Git Does During Merge

Step 1: Find Common Ancestor

$ git merge-base main feature-branch def456 ‚Üê Commit B

Git traces both branches back to find where they diverged.

Step 2: Three-Way Merge

  • Base: Common ancestor (B)
  • Ours: Current branch tip (F)
  • Theirs: Other branch tip (E)

üìã Step 3: Git's Merge Decision Logic ‚ñ∂
File State Base Ours (main) Theirs (feature) Result
Only main changed A B A ‚úÖ Use B (main's version)
Only feature changed A A C ‚úÖ Use C (feature's version)
Both changed same way A B B ‚úÖ Use B (identical)
Both changed differently A B C ‚ö†Ô∏è CONFLICT - manual resolution
File added on main - new - ‚úÖ Add file from main
File added on feature - - new ‚úÖ Add file from feature
Same file added both places - ver1 ver2 ‚ö†Ô∏è CONFLICT if content differs

Step 4: Create Merge Commit

# Git creates a new tree with merged contents $ git write-tree abc123def456... ‚Üê New tree hash (merged state) # Git creates merge commit with TWO parents $ git commit-tree abc123def456... -p pqr678... -p mno345... -m "Merge feature-branch into main" xyz999... ‚Üê New merge commit hash # Git updates main branch to point to merge commit $ echo "xyz999..." > .git/refs/heads/main

Fast-Forward Merge (No Merge Commit)

gitGraph commit id: "A" commit id: "B" branch feature commit id: "C" commit id: "D" checkout main merge feature id: "Fast-forward to D"

Fast-Forward: When main hasn't moved

If main hasn't changed since feature branched off, Git can simply move main's pointer forward. No merge commit needed!

# Before merge main ‚Üí B feature ‚Üí D (which has parent ‚Üí C ‚Üí B) # After fast-forward merge main ‚Üí D (just moved the pointer!)

üéØ Final Takeaways

  1. Commits store metadata: They don't store files directly, just pointers to trees
  2. Trees are directories: They list blobs (files) and other trees (subdirectories)
  3. Blobs are files: Actual file contents, stored once and reused via hashing
  4. Merge commits have multiple parents: This is how Git tracks branch history
  5. The tree in a merge commit: Shows the final state after resolving conflicts
  6. Git uses parent pointers: To traverse history, find common ancestors, and perform merges
