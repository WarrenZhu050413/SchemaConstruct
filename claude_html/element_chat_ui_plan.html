üìù Duplicate Prompt Rendering ‚ñ∂
Investigation
  • User reports double ‚ÄúYou‚Äù bubbles per send; likely stemming from redundant handleSendMessage/sendMessageToAPI invocations (lines ~646 & ~426) or queue replay.
  • No tests cover this; need instrumentation to track `queued-${timestamp}` IDs and confirm single dispatch.
  • Chrome storage writes via elementChatService.addMessageToChat may race with `saveSession` debounce, risking rehydration merges.
Plan
  • Add temporary console counters (or dev-only telemetry) around handleSendMessage, processQueue, and `setMessages` to capture duplicate flows.
  • Guard `handleSendMessage` with an `isSubmitting` ref that blocks re-entry until the async `sendMessageToAPI` resolves, ensuring queue usage is explicit.
  • Within `sendMessageToAPI`, ensure we use the same generated ID for storage and UI append, and de-dupe by ID before `setMessages` (e.g., map by `message.id`).
  • After fix, craft a Playwright regression: send a prompt, assert only one `.role-user` bubble, then send again while stream active to verify queue path.
Validation

Playwright e2e: send rapid double-enter; snapshot conversation showing single user entries.

‚ÜîÔ∏è Resizable Chat Window ‚ñ∂
Investigation
  • ElementChatWindow already uses react-rnd with `enableResizing`, but the parent container (`pointer-events: none`) may block handles.
  • No visual affordance for handles; users likely miss the interactive edges.
  • Resize persistence relies on `handleResizeStop` (lines ~705) but lacks an `onResize` handler, so window may snap back during drag.
Plan
  • Set `style={{ pointerEvents: 'auto' }}` on the Rnd wrapper to ensure handles receive events despite the host container.
  • Add custom `resizeHandleComponent` with red corner grips (bottom-right, right, bottom) sized ~12px for visibility.
  • Implement `onResize` to live-update `windowSize` so the content follows cursor, still persisting final size on `onResizeStop`.
  • Verify `minHeight`/`minWidth` allow comfortable layout (e.g., 320√ó280) and persist to `session.windowState` through `saveSession`.
Validation
Manual: drag handles on a complex page, ensure no jitter and state reload restores custom size.
Playwright: mutate size, close/reopen chat, assert new dimensions via DOM bounding box.
Cross-check queue panel & image upload UI scales correctly in condensed widths.
üîî Persistent Element Indicators ‚ñ∂
Investigation
  • Current üí¨ badge exists only inside ElementSelector overlay; once the chat window closes, elements have no visual cue.
  • Sessions can bind to multiple descriptors (multi-anchor); indicator must reflect all anchors.
  • Chat history clears via `handleClearHistory` but we do not broadcast state changes to remove badges.
Plan
  • Create elementChatIndicatorService managing badge elements attached to the page (e.g., absolutely positioned spans with red dot icon).
  • On content script init, call `getAllElementChats(window.location.href)` and render badges for each descriptor using stable `chatId` targets.
  • Expose `registerIndicator` / `removeIndicator` helpers; invoke them when chats open, when `saveSession` persists messages, and when history is cleared or session deleted.
  • Ensure badges survive DOM mutations by observing target relocation (reuse existing `ensureAnchorPosition` logic or MutationObserver).
Validation
Manual: open chat, close window, confirm badge remains; clear history ‚Üí badge disappears.
Playwright: multi-anchor chat should render badges on each linked element.
Performance check: ensure MutationObserver throttled to avoid excessive reflows on dynamic pages.
Design badge visuals to avoid overlapping interactive controls; consider offsetting using bounding rect + 8px padding.
